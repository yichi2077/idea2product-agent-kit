**`idea2product-agent-kit` 是 [Anthropic《创始人手册》(Founder's Playbook)](https://claude.com/blog/the-founders-playbook) 精神的本地化、强制化、对话内的工程落地。** 它把手册的核心纪律——*先验证，再构建*——固化成一台确定性的「阶段-审批门」状态机:你的 AI 编码代理(Claude Code、Cursor、Codex…)**在对话框内**驱动它,在想法、战略、产品与架构通过验证并获批之前,**硬性拦截过早写代码**。

> *"42% 的创业公司失败,是因为做了没人想要的东西——而 AI 让这种失败模式更容易发生,不是更难,因为一个能跑的原型太容易被误当成'已验证'了。"* —— 这正是《创始人手册》点明的问题,也是本 kit 为之而生要去阻止的事。

## 为什么需要它

在 AI 时代,产品死掉的原因不是「做不出来」,而是「做了个没人要的东西」。智能体编码把「想法 → 能跑的原型」压缩到一个下午,这反而让人**更容易**跳过唯一重要的事:证明问题是真实的。

放任不管,AI 编码代理会:
1. 拿到原始需求就立刻开始大量生成代码;
2. 重复造已有的轮子,并遗漏关键的业务与安全约束;
3. 在没有产品定义和架构作为锚点的情况下,前期迅速累积技术债;
4. 把自己的上下文窗口拖垮,陷入无尽的重构循环与碎片化代码库。

本 kit 安装的是一套 **「左移 (Shift-Left)」** 纪律:在动任何 **怎么做 (How)** 之前,先**带着证据**把 **为什么做 (Why)** 与 **做什么 (What)** 厘清。它刻意做到 **代理无关 (agent-agnostic)** 且 **本地优先 (local-first)**:一台轻量级 Python 状态机(`pipeline.py`)加上提交进你仓库的 Markdown 产物。无 SaaS,无云端锁定。

## 基于《创始人手册》构建

本 kit 不只是**引用**《创始人手册》,而是把它的 **Idea + MVP 方法论作为强制工作流落地**。手册是 AI 可以悄悄忽略的散文式建议;本 kit 把这些建议变成**结构性**的东西:强制步骤、人工审批门,以及引擎自己会跑的检查。

| 《创始人手册》原则 | 本 kit 如何强制落地 |
| :--- | :--- |
| 把想法变成**可证伪的假设** | **P1** 构思备忘:人群 · 频率 · 量化损失 · 现有变通方案 |
| **绘制竞争格局**(别对竞争对手视而不见) | **P2** 竞品扫描:四层(直接/间接/潜在收购方/邻近) + 竞品**差评挖掘** + **steelman 钢人法**(替最强对手辩护) |
| 做**客户探索访谈**——问**过去的真实行为**,而非未来意愿 | **P4** 访谈计划:防诱导的提问清单、逐次访谈复盘、每 5 次的**不对称预警** |
| 把 AI 当作**每个阶段的「结构化反方辩手」** | **P2–P4** 强制红队/事前验尸、**P5** 全新上下文的 PM 评审、**P7** 架构红队 |
| *"确认偏误现在配上了一台研究引擎"* | **证据来源(provenance)字段** + **确认偏误护栏**:`doctor` 与每道门都会标记「无用户接触就关闭」「零反证就关闭」「整体证据偏向支持」的假设 |
| *"原型不是证据——对话才是"* | **P6 验证原型**:做出单一核心交互,交给 ~5 位目标用户实测,在投入完整构建**之前**决定 继续/转向/豁免 |
| 守住 MVP 范围——*"是真的用户信号,还是创始人的一厢情愿?"* | **P5** PRD:明确的**非目标** + **证据门槛的加功能准则** |
| **在任何用户接触之前先做一次安全审查** | **P9** 推荐的发布前安全审查(鉴权、信息泄露、注入、依赖 CVE) |
| 朝着**证据(PMF)**迭代,而非朝着完美 | **P6 / P10** 的 PMF 工具:Sean Ellis 40% · 努力测试(推 vs 被拉) · 三个 R,外加转向诊断 |

> **范围声明(诚实地说):** 本 kit 完整落地了手册的 **Idea** 与 **MVP** 方法论。手册的 **Launch / Scale** 内容大多是*运营*(创始人精力分配、增长团队搭建、合规项目化),刻意**不纳入范围**——本 kit 是一条 0→1 的「验证-构建」流水线,而非运营平台。完整的映射、取舍理由与审计记录见 [`docs/founder-playbook-review/`](docs/founder-playbook-review/)。

## 工作原理:阶段-审批门状态机

工作区被划分为 **10 个线性阶段** 和 **4 道人工审批门**。在每道门获批之前,引擎拒绝推进——尤其拒绝让代理写产品代码。

```
     为什么做? (WHY)            做什么? (WHAT)               怎么做? (HOW)
┌─────────────────┐     ┌─────────────────────┐     ┌─────────────────────┐
│  P1 – P3        │────▶│  P4 – P6            │────▶│  P7 – P10           │
│  战略研究与        │     │  产品定义与          │     │  技术架构、           │
│  可行性决策       │     │  验证               │     │  特性 Specs 与构建  │
└─────────────────┘     └─────────────────────┘     └─────────────────────┘
```

| 阶段 | 描述 | 主要产出 | 审批门 |
|-------|------|---------|--------|
| **P1** | 构思备忘 | 可证伪假设、不做的代价、停止条件 | — |
| **P2** | 战略调研 | 竞品扫描(用/买/合/建)、市场分析、风险登记册 | — |
| **P3** | 战略决策 | 自建/外购/合作备忘录、战略红队、护城河检查 | **战略之门** |
| **P4** | 产品发现 | JTBD、机会树、精益画布、**访谈计划** | — |
| **P5** | 产品定义 | PRD(非目标 + 证据门槛范围)、用户故事、PM 评审 | **产品之门** |
| **P6** | **验证原型** | 单一核心交互 + ~5 位用户实测;继续/转向/豁免 *(可豁免)* | — |
| **P7** | 架构交付 | ADR、技术 Spike、特性地图、追溯矩阵 | **架构之门** |
| **P8** | 特性规约 | 测试驱动、Spec-Kit 就绪的特性规格包 | — |
| **P9** | 构建发布 | 经验证的实现、安全审查、发布决策 | **发布之门** |
| **P10** | 效果复盘 | 投产后假设验证、PMF 信号、转向/坚持 | — |

整条流程都**在代理对话框内**驱动:打字说 `run p1` 或 `next`,代理就会运行引擎、遵循阶段配方、生成 Markdown 产物。唯一需要人类亲自动手的,只有「批准一道门」。

## 核心差异化优势

- 🧭 **用证据压倒热情。** 持久化的**假设登记册**记录每条主张「如何被验证」(真实用户?实验?案头研究?)。引擎的**确认偏误护栏**会在过门前标记自欺——这正是散文式建议替你做不到的事。
- 🧪 **一个真正的验证步骤,不只是原型。** P6 把手册最难的一课塞进工作流:做一个交互、放到真人面前,让**他们的行为**来决定继续还是转向——在你为完整构建砸下数周之前。
- 🛑 **确定性,而非自主放飞。** 带人工审批门的状态机——没有失控的代理循环,没有「AI 审 AI」。
- 🧱 **极低上下文开销。** 代理永远只加载当前活跃阶段及其交付物,长项目下上下文不崩。
- 🔒 **本地优先,零 SaaS。** YAML 状态 + Markdown 报告都在你的 Git 仓库里,外部服务看不到你的想法。
- 🤖 **代理无关、技能化(skills-based)。** 兼容 Claude Code、Cursor、Codex、OpenCode、Hermes、OpenClaw 及任意 AGENTS.md 代理——全程通过对话内的 skill / 自然语言完成。
- 📋 **代理的「信息真理源」。** P1–P5 的产物在构建阶段成为代理的事实依据,大幅削减幻觉与逻辑漂移。

## 横向对比

| 维度 | `idea2product` | 裸用 Agent (Aider / Claude Code / Cursor) | 多智能体框架 (CrewAI / AutoGen) |
| :--- | :--- | :--- | :--- |
| **执行路径** | 确定性状态机;**先文档,后代码** | 即兴生成;先写代码,规格随缘 | 自主循环;漂移风险高 |
| **验证** | **证据可追溯** + 确认偏误护栏 | 无——纯凭感觉 | 自我校验(AI 审 AI) |
| **上下文** | 低——只看当前阶段 | 高——混合历史拖垮上下文 | 极高——多代理对话烧 Token |
| **质量控制** | 人工审批门,留痕 | 事后人工审查 | 常漏边缘情况 |
| **隐私与成本** | 本地优先、零 SaaS | 本地优先 | 常依赖云端、API 成本高 |

## 适用人群与场景

**最适合**
- **独立开发者与 Indie Hackers** —— 阻止你为未经用户定位(P4)、未经单位经济学(P2)验证的伪需求写代码。
- **技术产品经理** —— 在本地生成规范的 PRD、验收标准与架构 handoff,可直接交付给开发团队。
- **团队技术负责人 (TL)** —— 在 AI 协作中强制落地设计优先、TDD 的工程文化。

**适用范围:** 0→1 新项目 · 大型 1→N 特性迭代(支付、OAuth…) · 技术探路与可行性研究(P7)。

**不适用:** 紧急修复与微调(为改个错别字走 10 阶段流水线得不偿失) · 高度耦合的遗留单体 · 全自动无人值守的代码生成。

## 安装与快速上手

**前置条件:** Python 3.10+(仅标准库) · Git · 兼容的编码代理。

### 一键上手(推荐)
```bash
git clone https://github.com/yichi2077/idea2product-agent-kit.git

# 安装技能 + 初始化引擎 + 接线宿主适配器(自动检测)
python3 idea2product-agent-kit/scripts/install.py init /path/to/your-project
```

### 或分步进行
```bash
# 在目标项目中初始化 .pipeline 引擎 + docs
python3 idea2product-agent-kit/scripts/install.py scaffold /path/to/your-project

# 安装代理适配器(例如 Claude Code)
python3 idea2product-agent-kit/scripts/install.py adapters /path/to/your-project --agent claude-code
```
*Cursor、Codex、OpenCode、Hermes、OpenClaw 或通用 AGENTS.md 代理,请参考 [技术操作手册(英文)](docs/TECHNICAL-DEEP-DIVE.md#1-installation-details-per-agent)。*

### 启动
在代理对话框里直接打字:
```
run p1     (或: next)
```
代理会询问你的想法,并带你走完整条流水线。一切都在对话里完成。

## 核心机制

### 人工审批门
推进会被拦截,直到审批被记录进 `.pipeline/state/pipeline-state.yaml`。
- **Light 模式(默认):** 在对话框里审阅文档,然后让代理批准(例如:*"approve the strategy gate — economics validated"*),代理负责登记。
- **Strict 模式:** 代理不能自批;引擎要求你在独立终端里输入随机挑战码。
  ```bash
  python3 .pipeline/scripts/pipeline.py gate mode strict
  ```

### 确认偏误护栏
在 `doctor` 和每道门处,当某条假设「无用户接触就关闭」「零反证就关闭」,或登记册整体偏向支持性证据时,引擎都会给出**非阻塞**预警——让你在投入前权衡的是真实证据,而非一厢情愿。

### 集成诊断
```bash
python3 .pipeline/scripts/pipeline.py doctor     # 状态一致性、缺失交付物、偏误护栏
python3 .pipeline/scripts/pipeline.py status     # 当前阶段、门状态、阻塞项
python3 .pipeline/scripts/pipeline.py handoff    # 综合简报:决策、开放假设/风险
```

### Git 审计追踪
每个阶段结束时,代理会用统一信息(如 `idea2product: complete P2`)提交文档与状态,留下干净可审查的历史。绝不自动 push。

## 参与贡献

1. **Fork** 仓库并创建特性分支(`git checkout -b feature/amazing-feature`)。
2. 完成修改并写清说明。
3. 提 PR 前,运行完整性检查:
   ```bash
   python3 scripts/sync_bundled_copies.py --check
   python3 scripts/check_skill_refs.py
   ```
4. **推送**并提交 **Pull Request**。较大的改动,请先开 Issue 讨论。
